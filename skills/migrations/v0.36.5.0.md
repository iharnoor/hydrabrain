# v0.36.5.0 — Secure DATABASE_URL access for shell jobs calling `gbrain` CLI

## What changed

- **Shell-job pre-enqueue validator** (`src/core/minions/handlers/shell-validate.ts`,
  NEW). Called from BOTH submit surfaces BEFORE `MinionQueue.add()`:
  `gbrain jobs submit shell` (CLI) and `submit_job` MCP op for `name='shell'`.
  Validates cmd/argv/cwd/env shape, the new `inherit:` closed enum, env-shadow
  rejection, and fail-fast on missing config. No rejected payload ever lands
  in `minion_jobs.data`.
- **`inherit: ["database_url"]` ShellJobParams field** with a single-source
  `INHERITABLE` record (`shell-inherit.ts`). Worker resolves the secret from
  its own `loadConfig()` at child-spawn time, injects under `GBRAIN_DATABASE_URL`.
  Names persist in the row + audit; values never do.
- **`env: { GBRAIN_DATABASE_URL: ... }` and `env: { DATABASE_URL: ... }` are
  rejected at validation** regardless of whether `inherit:` was passed.
  Error message points at `inherit: ["database_url"]` as the replacement.
- **Defense-in-depth re-validation** in the shell handler at job-pickup catches
  pre-existing rows from before v0.36.5.0 AND any future submit path that
  forgets to call the pre-enqueue validator.
- **`gbrain doctor home_dir_in_worktree` check** warns when `~/.gbrain/` lives
  inside a git worktree (Conductor + worktrees topology). Handles `.git` as
  directory or file (linked worktree). Walk terminates at `$HOME`.
- **`~/.gbrain/.gitignore` retroactive** via `ensureGitignore()` called from
  `saveConfig()` AND `gbrain post-upgrade`. Idempotent, never clobbers user
  content. Honest scope: blocks casual `git add`, NOT already-tracked files,
  screenshots, backups, or `git add -f`.
- New canonical guide: `docs/guides/agent-to-gbrain.md`. Two-domain framing
  for downstream agent authors: MCP ops via thin-client OAuth vs `localOnly`
  admin ops via shell-job `inherit:`.

## What downstream agents must do

If your agent submits shell jobs that call `gbrain` CLI, update the params.
Pre-v0.36.5.0 workarounds documented in PR #1137 (now closed) are rejected at
submit time as of v0.36.5.0.

### Before (REJECTED)

```jsonc
{
  "cmd": "gbrain sync --skip-failed",
  "cwd": "/data/gbrain",
  "env": { "GBRAIN_DATABASE_URL": "postgresql://..." }
}
```

### After

```jsonc
{
  "cmd": "gbrain sync --skip-failed",
  "cwd": "/data/gbrain",
  "inherit": ["database_url"]
}
```

The worker host must have `database_url` configured (one-time operator action):

- `gbrain config set database_url postgresql://...` (file plane), OR
- `GBRAIN_DATABASE_URL` / `DATABASE_URL` env on the worker process.

If the worker can't resolve `database_url`, the validator rejects the job at
submit time with a paste-ready `gbrain config set database_url` hint.

## Verify

```bash
# 1. The new pattern works on your worker:
gbrain jobs submit shell --params \
  '{"cmd":"gbrain stats","cwd":"/tmp","inherit":["database_url"]}' --follow
# Expect: page count, exit 0.

# 2. The doctor surfaces worktree risk if it exists:
gbrain doctor --json | grep -A1 home_dir_in_worktree

# 3. The retroactive gitignore landed:
test -f ~/.gbrain/.gitignore && cat ~/.gbrain/.gitignore
# Expect: file exists, contents = "*\n"

# 4. The audit-log records names not values:
tail -1 ~/.gbrain/audit/shell-jobs-*.jsonl | grep -o '"inherit":\[[^]]*\]'
# Expect: ["database_url"]
```

## Scope this PR

`inherit: ["database_url"]` only. Provider API keys (Anthropic, OpenAI, Groq,
Voyage) and the remote-MCP client secret are NOT in the inherit allowlist in
v0.36.5.0. The worker's gateway resolves provider keys at startup from its
own env or config; shell jobs calling `gbrain` rely on that. A follow-up PR
will do the `GBrainConfig` + `buildGatewayConfig()` refactor needed to add
provider keys properly.

## Why this exists

PR #1137 (PR #1137 author) documented two workarounds for the env-stripping
behavior of shell jobs. Both worked. Both left plaintext secrets where they
shouldn't be — either in `~/.gbrain/config.json` on disk OR in
`minion_jobs.data` rows that travel with brain DB dumps and shared brains.

The `/cso` audit cut an earlier proposal (encrypted vault + Unix-socket
broker + SO_PEERCRED + per-call tokens) as theater for a single-uid topology.
The codex outside-voice pass caught the load-bearing bug in the rewritten
"minimum scope" plan: validation in the handler runs AFTER `queue.add()`,
so the headline "R1 closed" was technically false. Fixed pre-implementation
by lifting the validator to a shared pre-enqueue module called from both
submit surfaces.
