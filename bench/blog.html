I caught my own benchmark lying about HydraDB

I forked an open-source agent-memory engine, rebuilt its retrieval layer on HydraDB, and ran a head-to-head. The first numbers were a blowout. They were also a lie — and the lie was mine.

By Harnoor 2026-06-23 HydraDB engineering

I set out to prove that HydraDB — a hosted, graph-native store for agent memory — beats the local-first alternatives. The obvious opponent was gbrain, garrytan's open-source agent-memory engine. It's well-built, widely used, and explicitly aims to be "the next Postgres for memory." A fair fight against a serious system.

So I forked it. I kept gbrain's skills, its operations contract, its ingestion pipeline — and swapped its retrieval layer for HydraDB. Same prose in, two memory engines out the back. Then I wrote a harness, hit run, and watched HydraDB win everything.

That should have felt good. It felt wrong. When your own benchmark hands you a clean sweep against a system you respect, the most likely explanation isn't that you're a genius. It's that you cheated without noticing.

01The blowout was a lie

I went back through my own harness looking for the seams. I found three. Every one of them quietly disadvantaged gbrain, and together they meant I had been benchmarking gbrain with its headline feature switched off.

  • BUG 1 — gbrain's graph was never built. gbrain's typed-edge graph is one of its best ideas: it infers relationships from prose and lets you traverse them. But the default schema pack I was using shipped no edge-inference regexes. Result: zero edges. I was scoring a graph engine that had no graph.
  • BUG 2 — when edges did build, they were mis-typed. Once I wired in extraction, "person-00 works at company-00" came back labeled founded instead of works_at. Wrong relationship type means the traversal answers the wrong question. Garbage edges are worse than no edges.
  • BUG 3 — gbrain's relational arm was dormant by default. Even with a correct graph, gbrain's query path was vector-only unless you explicitly pass relational:true. I hadn't. So the graph I'd fought to build was sitting there unused at query time.

None of these are gbrain's fault. They're defaults — reasonable ones — and they were my responsibility to configure correctly before claiming anything. A benchmark that measures a competitor with their main feature dark isn't a benchmark. It's a press release with a chart.

02Rebuilding it fair

I threw the first harness out and rebuilt it to be honest by construction:

  • Identical input. The same synthetic prose goes into both engines. No engine gets a cleaner copy of the corpus.
  • An integrity gate. The harness now aborts if gbrain's edges are missing or mis-typed. If the graph isn't real, no scores print. You can't accidentally re-run the original lie.
  • gbrain's relational arm engaged. Queries run with relational:true so gbrain's traversal actually fires.
  • And then I went further. To kill the obvious rebuttal — "you broke gbrain's extraction, of course it lost" — I handed gbrain a 100%-correct graph using its own published seeding method. Perfect edges, perfect types, no extraction error to blame. Best-case gbrain versus HydraDB.
On the corpus

Everything here runs on a small, synthetic corpus: 50 entities, 149 queries. Generic placeholders only — person-00, company-00, fund-a. This is not gbrain's published 240-document scale, and I won't pretend it is. It's big enough to expose retrieval behavior on multi-hop relational questions; it's small enough that you should read every number below as a signal, not a verdict.

Read that table honestly, because the honest reading is the whole point. gbrain wins the simple 1-hop lookups outright — 100.0% to HydraDB's 92.3%. When the answer is one edge away and the graph is perfect, gbrain's deterministic traversal is flawless and HydraDB occasionally misses. I'm stating that plainly first, because everything else I'm about to claim is only believable if I'm willing to say where I lose.

What HydraDB does win is the part that actually gets hard: multi-hop traversal. On 2-hop questions — "which fund backed the company that person-00 works at" — HydraDB holds 86.0% recall against gbrain's 63.8%, even with gbrain's graph hand-built to perfection. Chaining relationships is where retrieval systems break, and it's where the gap opens.

04The architecture argument

The retrieval-quality numbers are close enough to argue about. The architecture isn't. This is where HydraDB's case is clearest, and it's not about a metric — it's about how much machinery you have to operate to get there.

DimensionHydraDBgbrain
Lines of retrieval code3279,345
Files132
Query pathone recall() call6-stage pipeline

gbrain's six stages — dense vectors, BM25, reciprocal-rank fusion, a reranker, query expansion, and the graph arm — are each individually reasonable engineering. Stacked, they're roughly 29× the retrieval code across 32 files and four services you have to keep alive. HydraDB folds all of that behind a single recall(). The result is comparable on quality and dramatically smaller to operate, verify, and reason about.

05At scale — and a caveat I won't bury

The synthetic corpus is small. To stress retrieval under real distractor pressure I ran LongMemEval-S (ICLR 2025), which buries each answer under roughly 48 distractor sessions per question.

The 16.7% opponent here is a BM25 lexical baseline, not gbrain. I have not run gbrain at LongMemEval scale, so this table says nothing about HydraDB-vs-gbrain. It says HydraDB's recall surfaces the right evidence (a tie at 97.6%) and then turns that evidence into correct answers far more often than naive lexical retrieval does. That's a real result — just not the one a careless skim would claim.

06What I actually believe now

HydraDB does not win every metric, and the version of this post that claimed it did would have been the lie I caught in step one. Here's the defensible version:

  • Robust auto-extraction from prose. When neither system gets a hand-built graph, HydraDB recovers far more of the relationships that are actually in the text.
  • Multi-hop traversal. The gap widens exactly where it's hard to win — chaining relationships — and it holds even when gbrain is handed a perfect graph.
  • Radical simplicity. One call, one service, ~29× less code, for comparable quality. That's the operational argument, and it's the strongest one.
  • And gbrain genuinely wins 1-hop. If your workload is simple single-edge lookups and you'll maintain a clean graph, gbrain's local-first determinism is excellent.

The fine print I owe you

  • HydraDB is a hosted service. It's non-deterministic — numbers drift a few points between runs — and it carries uptime and cost risk that gbrain's local-first store simply doesn't. If you need an air-gapped, fully reproducible store on your own disk, that's a real point in gbrain's favor.
  • The corpus is small and synthetic (50 entities, 149 queries), well below gbrain's published 240-doc scale. Treat the relational tables as directional.
  • Every "win" above is "win on this corpus, at this scale, on this day." Rerun it and the third decimal will move.

Reproduce it

The only benchmark worth anything is one you can run yourself. This is the whole thing — fairness gate, relational arm, optional perfect-graph seeding — in one command:

# fair relational benchmark, with gbrain handed a perfect graph
python3 -m bench.relational_v2 --seed-edges --report

If gbrain's edges aren't built and correctly typed, it aborts before printing a single number. That gate is there because the first version of this post would have been wrong without it — and because I'd rather you catch me than have to catch myself twice.

HydraDB is a hosted, graph-native agent-memory store — hydradb.com. gbrain is open-source agent memory by garrytan — github.com/garrytan/gbrain. All entities in the benchmark corpus are synthetic placeholders. This is a fair technical comparison, not a dunk: gbrain is a serious, well-built system, and the integrity of the result depends on being generous to it.
